# Plan
## Goal
既存の FastAPI TODO API をブラウザから操作できる、モダンでレスポンシブな UI を追加する。TODO の作成、一覧表示、完了状態の切り替え、削除を直感的に行え、ローディング、空状態、入力エラー、通信エラーが利用者に明確に伝わる画面を提供する。既存の `/todos` API 契約と Swagger UI は維持する。

## Tasks
1. UI の構成とデザイン方針を定義する。
   - 1 画面の TODO 管理 UI とし、ヘッダー、作成フォーム、件数サマリー、フィルター、TODO リスト、通知領域で構成する。
   - 色、余白、角丸、影、タイポグラフィ、フォーカス表示を CSS カスタムプロパティとして整理し、一貫したデザインにする。
   - モバイルを基準にしつつ、タブレットおよびデスクトップで読みやすい最大幅とレイアウトを設定する。
2. FastAPI からフロントエンドを配信する。
   - `app/static/` に HTML、CSS、JavaScript を配置し、`app/main.py` で静的アセットを配信する。
   - `GET /` で UI の HTML を返し、既存の `/todos`、`/docs`、`/openapi.json` と競合しないルーティングにする。
   - 外部 CDN を必須にせず、ローカル起動だけで表示できる構成にする。
3. TODO 操作を UI に実装する。
   - 初回表示時に `GET /todos` を呼び出して一覧を描画する。
   - 作成フォームから `POST /todos` を実行し、成功時は入力をクリアして一覧と件数を更新する。
   - チェック操作で既存の `PUT /todos/{todo_id}` を使い、タイトルを保持したまま完了状態を切り替える。
   - 削除操作で `DELETE /todos/{todo_id}` を実行し、成功時に対象項目を一覧から除く。
   - 「すべて」「未完了」「完了」のクライアント側フィルターと、総件数・未完了件数の表示を追加する。
4. 操作状態とエラー処理を整備する。
   - 初回取得中のローディング表示、TODO がない場合およびフィルター結果がない場合の空状態を用意する。
   - 送信中は多重操作を防止し、対象のボタンやフォームに処理中であることを示す。
   - API の 422、404、通信失敗を利用者向けメッセージとして通知し、失敗した更新では画面上の状態を API の状態と不整合にしない。
   - タイトルの前後空白を考慮し、空文字と 200 文字超過をクライアント側でも検証する。ただし API 側の検証を最終的な正とする。
5. アクセシビリティと操作性を確保する。
   - フォーム要素にラベルを関連付け、キーボードのみでも作成、フィルター、完了切り替え、削除を操作可能にする。
   - フォーカスリング、十分な色コントラスト、ボタンの操作領域を確保し、色だけに依存しない状態表現を使う。
   - 通知領域には `aria-live`、処理中の領域には適切な `aria-busy` を設定し、完了済み項目の状態を支援技術へ伝える。
   - OS の `prefers-reduced-motion` を尊重し、動きを抑える設定では不要なトランジションを無効化する。
6. テストとドキュメントを更新する。
   - `GET /` が HTML を返すことと、必要な静的アセットが取得できることを pytest で検証する。
   - 既存 API テストを実行し、UI 配信の追加によって CRUD 契約が回帰していないことを確認する。
   - README に UI の URL、基本操作、画面確認手順を追記する。
   - ブラウザで主要操作、狭い画面幅、キーボード操作、ローディング・空状態・エラー状態を手動確認する。

## Acceptance Criteria
- `uvicorn app.main:app --reload` で起動後、`GET /` にアクセスすると追加のビルド手順なしで TODO UI が表示される。
- UI から TODO の作成、一覧確認、完了・未完了の切り替え、削除ができ、再取得した API の内容と一致する。
- すべて・未完了・完了のフィルターと件数表示が、作成、状態変更、削除の直後に正しく更新される。
- 空の一覧、フィルター結果なし、ローディング中、入力エラー、API エラーがそれぞれ判別可能な表示になる。
- 320px 程度の狭い画面からデスクトップまで横スクロールなしで主要操作が行え、キーボード操作時にフォーカス位置が視認できる。
- UI のフォームと操作ボタンに識別可能なラベルがあり、通知と処理中状態が支援技術へ伝達される。
- 既存の `/todos` API、`/docs`、`/openapi.json` が従来どおり利用でき、既存テストと追加した UI 配信テストがすべて成功する。
- README に記載された手順だけで UI を起動して確認できる。

## Risks
- 「モダン」の解釈は主観的で、ブランドカラーや参考デザインが未指定である。実装前に配色・雰囲気の合意が得られない場合は、ニュートラルな配色、明確な階層、控えめな影とアニメーションを初期方針とする。
- 現在の API は部分更新用の `PATCH` を持たないため、完了状態の変更でもタイトルを含む `PUT` が必要になる。クライアントは現在値を保持して全フィールドを送信し、将来 API が拡張された場合に切り替える。
- インメモリストアのため、再起動で TODO が消え、複数ワーカーでは表示と更新の整合性を保証できない。この制約を README と空状態で誤解なく伝える。
- JavaScript 無効時には動的操作が利用できない。今回は既存 JSON API を利用する軽量クライアントを対象とし、サーバーサイドフォーム対応は対象外とする。
- 自動テストだけではレスポンシブ表示、コントラスト、支援技術での読み上げを完全には保証できないため、代表的な画面幅とキーボード操作の手動確認を完了条件に含める。
