# Plan
## Goal
FastAPI を使用した TODO API を新規構築し、TODO の作成・一覧取得・個別取得・更新・削除を提供する。pytest による自動テストと、セットアップ、起動、テスト、API 利用方法を説明する README を含める。

## Tasks
1. プロジェクトの基本構成と依存関係を定義する。
   - FastAPI アプリケーション、テスト、依存関係定義ファイルを配置する。
   - 実行依存関係として FastAPI と ASGI サーバー、開発依存関係として pytest と HTTP テストクライアントを追加する。
2. TODO のデータモデルと API 契約を定義する。
   - TODO は `id`、`title`、`completed` を持つものとし、作成用・更新用・レスポンス用の Pydantic モデルを用意する。
   - 空タイトルなどの不正な入力をバリデーションで拒否する。
   - 永続化方式は外部サービスを必要としないインメモリストアとし、ID を一意に採番する。
3. CRUD エンドポイントを実装する。
   - `POST /todos` で TODO を作成し、HTTP 201 と作成結果を返す。
   - `GET /todos` で TODO 一覧を返す。
   - `GET /todos/{todo_id}` で指定 TODO を返す。
   - `PUT /todos/{todo_id}` で指定 TODO を更新する。
   - `DELETE /todos/{todo_id}` で指定 TODO を削除し、HTTP 204 を返す。
   - 存在しない ID への個別取得・更新・削除は HTTP 404 とする。
4. pytest による API テストを追加する。
   - 各テストを独立させるため、テスト間でインメモリストアを初期化する fixture を用意する。
   - CRUD の正常系、404、入力バリデーション、ステータスコード、レスポンス内容を検証する。
5. README を作成する。
   - 前提環境、依存関係のインストール、開発サーバーの起動、pytest の実行方法を記載する。
   - 各 CRUD エンドポイント、リクエスト・レスポンス例、インメモリのため再起動時にデータが消える制約を説明する。
6. 品質確認を行う。
   - pytest を実行し、全テストが成功することを確認する。
   - README の手順でアプリが起動し、OpenAPI ドキュメントと主要エンドポイントへアクセスできることを確認する。

## Acceptance Criteria
- FastAPI アプリケーションがローカルでエラーなく起動する。
- TODO の作成、一覧取得、個別取得、更新、削除が定義された HTTP メソッドとパスで動作する。
- 作成された TODO には一意な ID が付与され、`title` と `completed` がレスポンスに含まれる。
- 不正な入力は HTTP 422、存在しない TODO への操作は HTTP 404 を返す。
- 削除成功時は HTTP 204 を返し、その後は対象 TODO を取得できない。
- pytest が CRUD の正常系および主要な異常系を網羅し、全テストが成功する。
- README の手順だけで環境構築、サーバー起動、テスト実行、API の基本操作ができる。

## Risks
- インメモリストアはプロセス再起動でデータが失われ、複数ワーカー間で状態を共有できない。本要件では簡潔さを優先し、永続化が必要になった場合はデータベースへ置き換える。
- グローバルな可変ストアはテスト間の状態漏れや並行更新の競合を招く可能性がある。テスト fixture による初期化を徹底し、本番利用時にはストレージ層の分離を検討する。
- 更新を `PUT` とするため、必須フィールドを含む全体更新として契約を明確にする。部分更新が必要な場合は別途 `PATCH` を追加する。
- FastAPI とテストクライアント周辺の依存バージョン不整合でテストが失敗する可能性があるため、互換性のある依存バージョンを定義する。
